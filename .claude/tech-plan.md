# 技术方案设计

主题：$ARGUMENTS

如果主题为空，先询问用户要设计什么主题的技术方案。

## 执行流程

按以下循环流程运行，最多 3 轮：

### 每轮执行步骤

**第一步：启动 Writer agent**

使用以下 prompt 启动 Writer agent（等待完成后再继续）：

```
你是一个资深技术方案设计专家。

任务：为「$ARGUMENTS」设计完整技术方案。

{{如果有审查反馈，附上：}}
上一版本审查意见如下，请针对性修改：
{FEEDBACK}
{{end}}

要求：
- 架构设计：系统整体架构、模块划分、组件交互
- 技术选型：核心技术栈选择及理由
- 实现步骤：分阶段的具体实施计划
- 风险评估：潜在风险及应对措施
- 方案要具体可落地，避免空泛描述

将完整方案写入文件 /tmp/plan.md，使用 Markdown 格式。
```

**第二步：启动 Reviewer agent**

Writer 完成后，使用以下 prompt 启动 Reviewer agent（等待完成后再继续）：

```
你是一个严格的技术方案审查专家。

请读取文件 /tmp/plan.md，从以下 5 个维度审查方案：
1. 可行性：方案是否技术上可实现，有无明显缺陷
2. 完整性：是否覆盖架构、选型、步骤、风险四个方面
3. 安全性：是否考虑了安全风险和防护措施
4. 性能：技术选型和架构是否能满足性能需求
5. 可维护性：方案是否易于后续维护和扩展

输出格式（写入 /tmp/review.md）：
- 若通过：第一行写 PASS，后面写各维度的通过理由
- 若不通过：第一行写 FAIL，后面用编号列出具体问题（每个问题说明所属维度和改进建议）
```

**第三步：读取审查结果**

读取 /tmp/review.md 第一行：
- 若是 `PASS`：跳转到「输出最终方案」
- 若是 `FAIL`：提取问题列表作为 FEEDBACK，进入下一轮

---

## 输出最终方案

审查通过后：
1. 读取 /tmp/plan.md 完整内容
2. 告知用户"方案已通过审查（第 N 轮）"
3. 展示最终方案全文

## 超出轮次处理

3 轮后仍未通过：
1. 展示 /tmp/plan.md 当前最新版本
2. 告知用户"已达最大轮次，以下是最终版本，仍存在以下问题："
3. 列出 /tmp/review.md 中的剩余问题，请用户决定是否继续修改
